# 霸业：190-208 历史扩容与战斗表现策划

## 目标

在不推翻当前可玩原型的前提下，把内容范围从“少量三国名将沙盘”扩展为“自 190 年诸侯讨董卓至 208 年赤壁战后”的低保真历史演化版本。

本阶段不追求一次性做完整三国剧本，而是先建立可持续扩容的数据模型、时间线入口和战斗表现框架，再按批次填充势力、城市、人物与事件。

## 当前问题

- 当前初始格局是架空混编：190 年日期与许昌、成都、建业三方并立并不一致。
- 当前仅有 4 座城市、4 个势力、12 名武将，无法承载 190-208 的主要历史转折。
- 当前武将只有 active、hidden、discovered 三种存在状态，没有按年月登场、归属切换、死亡退场机制。
- 当前战斗是一次性数值结算，玩家只能在出兵前做决定，战斗中没有参与感。

## 设计原则

### 历史范围

- 起点固定为 190 年 1 月，反董卓联盟形成后的局势。
- 首个完整历史阶段覆盖到 208 年 12 月，赤壁战后进入孙刘曹三方定型。
- 209 年后的荆州争夺、汉中与益州扩张只作为后续阶段预留，不在本批硬落地。

### 内容取舍

- 优先覆盖会改变势力版图、核心人才归属和大战节点的人物与城市。
- 不追求百科式全收录，优先做“玩家能感知差异”的君主、谋士、名将、关键据点。
- 城市先做核心节点，再补同州次级据点，避免地图过度膨胀压垮当前 UI。

### 实现节奏

- 先补数据模型与时间线触发，再补首批历史内容。
- 先补低保真战斗过程反馈，再决定是否继续下钻到兵种与战法层。
- 每批内容都以当前界面可承载为上限，避免在 origin/App.jsx 里继续堆条件分支。

## 历史阶段拆分

### S1 190-192 反董卓与关东混战

- 关键势力：董卓、袁绍、袁术、曹操、孙坚、公孙瓒、韩馥、刘表、吕布。
- 关键城市：洛阳、长安、陈留、邺、南阳、襄阳、北平、寿春、下邳。
- 关键体验：联盟讨董、迁都、董卓集团内斗、北方群雄割据开始成形。

### S2 193-199 群雄兼并与中原重组

- 关键势力：曹操、吕布、袁绍、袁术、刘备、张绣、刘表、孙策。
- 关键城市：许昌、濮阳、下邳、寿春、汝南、宛、徐州诸城、江东沿江据点。
- 关键体验：曹操迎帝、吕布覆灭、袁术僭号后崩解、孙策奠定江东基盘。

### S3 200-207 官渡后北方统一与荆州蓄势

- 关键势力：曹操、袁绍残部、刘备、刘表、孙权、马腾、张鲁。
- 关键城市：官渡战区、许昌、邺、襄阳、新野、江夏、建业、汉中。
- 关键体验：官渡决战、袁氏瓦解、刘备依附荆州、诸葛亮出山、曹操北征乌桓。

### S4 208 赤壁决战与三方定型

- 关键势力：曹操、孙权、刘备、刘表残部、刘璋、张鲁。
- 关键城市：襄阳、江陵、江夏、赤壁战区、夏口、柴桑、建业。
- 关键体验：荆州继承真空、孙刘联盟、长坂溃退、赤壁火攻、南郡争夺前夜。

## 首批历史扩容范围

### 势力

首批应从当前 4 势力提升到 10-12 个核心势力，至少覆盖：

- 玩家势力，占位为自定义诸侯，仍保留 sandbox 入口。
- 董卓集团。
- 曹操。
- 袁绍。
- 袁术。
- 刘备。
- 吕布。
- 刘表。
- 孙策/孙权集团。
- 公孙瓒。
- 马腾。
- 张鲁。

说明：玩家势力继续保留，是为了不把项目直接改成纯历史观察器。历史模式下玩家初始点位可配置为“联盟中一支自建势力”或接管指定君主，后续再决定是否开放剧本选择。

### 城市

首批城市建议扩至 14-18 座，优先覆盖：

- 司隶：洛阳、长安。
- 兖豫：陈留、许昌、濮阳、汝南。
- 冀青幽：邺、北平。
- 荆州：襄阳、江夏、宛、新野。
- 徐扬：下邳、寿春、建业、吴郡。
- 益雍预留：成都、汉中可作为后续阶段或锁定城市。

原则：城市数量先以“能让 190-208 主线转折发生”为标准，而不是按州郡全铺开。

### 人物

首批人物目标从 12 名提升到 45-60 名，分四层：

- 君主层：曹操、刘备、孙权、董卓、袁绍、袁术、吕布、刘表、孙策、公孙瓒、马腾、张鲁。
- 核心谋臣层：荀彧、郭嘉、贾诩、程昱、荀攸、诸葛亮、庞统、鲁肃、周瑜、张昭、法正预留。
- 核心名将层：夏侯惇、夏侯渊、曹仁、张辽、许褚、关羽、张飞、赵云、黄忠预留、甘宁、太史慈、周泰、吕蒙预留。
- 事件人物层：貂蝉、献帝、陈宫、田丰、沮授、刘琦、蔡瑁、张绣等。

原则：先补会影响外交、招募、攻城结果的人物，再补纯收藏型人物。

## 数据模型建议

### 武将字段

在现有武将结构上新增：

- debutYear：首次可登场年份。
- debutMonth：首次可登场月份。
- debutCityId：登场城市。
- debutFactionId：初始归属势力，可为 free。
- debutContext：登场背景，如继位、来投、出山、被发现。
- deathYear：死亡年份。
- deathMonth：死亡月份。
- deathType：死亡类型，如 illness、execution、battle、event。
- deathContext：死亡事件摘要。
- historicalImportance：用于事件抽样权重和 UI 标记。

说明：现有 hidden/discovered 仍可保留，但只作为“已进入世界后的可见性状态”，不能替代历史登场机制。

### 城市字段

建议新增：

- unlockYear、unlockMonth：该城首次进入地图主循环的时间。
- scenarioRole：如 capital、frontier、river-port、granary、pass。
- historicalControllerTimeline：用于记录关键年份归属变更模板。
- battleSceneTag：低保真战斗表现使用的地形标签，如 plain、fort、river、siege。

### 势力字段

建议新增：

- spawnYear、spawnMonth。
- spawnCityIds。
- collapseRules：君主死亡、主城陷落、历史事件后的解散或并入规则。
- historicalTags：如 coalition、northern-warlord、jingzhou、jiangdong。

### 时间线事件

新增独立历史事件数据源，按年月触发，事件类型至少包括：

- factionSpawn：势力登场或继承。
- officerSpawn：人物加入地图。
- officerTransfer：人物改投或被任用。
- officerDeath：人物死亡或处决。
- cityTransfer：脚本归属变动，只用于必须发生的关键节点。
- scriptedChoice：给玩家二选一的历史分歧点。

接入点应优先放在 monthlyTurnFlow，避免把时间线判断重新塞回界面层。

## 关键历史节点

首批必须覆盖的时间点：

- 190：关东联军起兵，董卓控制洛阳并迁都长安。
- 191：孙坚战死后，孙氏集团进入继承过渡。
- 192：董卓被诛，吕布与长安集团分裂。
- 196：曹操迎献帝都许，许昌成为中枢。
- 197：袁术僭号，声望与外交体系应受到重创。
- 199：吕布在下邳败亡。
- 200：官渡战役开始，袁曹力量分水岭形成。
- 202：袁绍病逝，袁氏进一步瓦解。
- 207：诸葛亮出山，郭嘉病逝。
- 208：刘表病逝，荆州继承分裂；曹操南征；孙刘联盟；赤壁决战。

## 数值与内容原则

### 君主与名将数值

- 曹操、周瑜、诸葛亮、郭嘉这类核心人物继续保持高辨识度，但需要避免单人决定一切。
- 同势力多名强将共存时，应更多通过任命槽位、同城协同、出战位限制来控制强度，而不是单纯压低数值。
- 190-199 阶段的人物流动性要更高，忠诚和归属变化应比后期更频繁。

### 城市经济与军事

- 许昌、邺、襄阳、建业应是核心增长节点。
- 洛阳在董卓迁都后可设计为高象征价值、低短期产出的争夺点。
- 江夏、下邳、宛这类前线城应更偏军事与事件密度，而非纯资源强城。

### 招募与探索

- 历史人物优先通过时间线登场或事件加入，而不是长期躺在 tavern 随机池里。
- 在野人才仍可保留少量探索发现，但要明确哪些是历史固定节点，哪些是随机人才。

## 低保真战斗表现方案

### 目标

- 让玩家在开战后仍有 2-3 次可感知决策。
- 不引入复杂战棋或实时系统，继续保持当前月度策略节奏。
- 复用现有统率、智力、士气、城防与外交破口数据。

### 表现结构

战斗拆为三段：

- 布阵阶段：展示双方兵力、主将、副将、地形、破口与初始士气。
- 交锋阶段：根据当前兵力和武将属性生成 2-3 个战况节点。
- 决胜阶段：显示胜负、战损、俘获、城防变化、名望影响。

### 玩家交互

每场战斗提供一轮主策略选择，条件允许时追加一轮临场应对：

- 稳扎：降低战损，降低爆发。
- 强攻：提高击破与夺城概率，增加粮草和兵员损耗。
- 奇袭：依赖智力、魅力、外交破口与地形，适合作弱胜强。
- 火攻：仅在 river 或 siege 场景中开放，受智力与天气事件加成。

说明：这套选择本质是对现有 battlePower 的分段修正，而不是另起一套复杂公式。

### UI 形态

- 使用日志卡片或步骤条展示三个阶段，不必单独开新页面。
- 关键数字变化只展示兵力、士气、城防、主将状态四类信息。
- 若存在外交破口，应在布阵阶段显式提示“内应响应”“守军迟疑”等文本。
- 胜败后保留一段可回顾的战报摘要，供玩家理解为什么打赢或打输。

### 技术落地建议

- 先把战斗从一次性结算改成“生成 battlePreview -> 玩家选策略 -> 解析 battlePhases -> 应用结果”的两步流程。
- battlePhases 先做纯数据结构，由界面顺序播放；不要先做动画。
- 复用 current attack action 的入口，但把公式与 phase 文案放入独立 engine 文件。

## 建议实施顺序

### P1 文档与模型

- 完成历史扩容策划文档。
- 设计历史人物、城市、势力、时间线字段。
- 规划 190 剧本首批势力与城市清单。

### P2 时间线基础设施

- 新增历史事件数据文件。
- 在 monthlyTurnFlow 接入按年月触发逻辑。
- 实现人物登场、死亡、改投的最小闭环。

### P3 190 剧本首批内容

- 把地图扩到首批核心城市。
- 补齐 10-12 个势力与 45-60 名人物。
- 为 190、196、199、200、208 五个关键年份补事件脚本。

### P4 低保真战斗过程

- 抽离战斗预演与阶段解析引擎。
- 新增战斗策略选择 UI。
- 为攻城和野战至少各提供一套阶段文案模板。

## 风险

- 如果直接在当前 gameConfig 上一次性堆几十名人物，后续历史登场与死亡机制会全面返工。
- 如果城市扩容过快而 UI 不改，现有列表式操作会明显恶化可读性。
- 如果战斗过程先做表现不做结构，origin/App.jsx 会再次膨胀，后续难以维护。

## 下一切片建议

下一轮实现优先做三件事：

- 为武将、城市、势力补充历史字段与类型约束。
- 新增历史时间线数据文件，并先接通 190、196、199 三个验证节点。
- 把当前战斗解析从 attack action 中抽到独立 engine，为后续战斗阶段 UI 预留数据结构。
